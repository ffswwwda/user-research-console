# 用户研究预测引擎 · User-Research Prediction Console

一个基于 [MiroFish](https://github.com/666ghj/MiroFish) 公开架构（OASIS / CAMEL-AI 多智能体社会仿真）改造的**用户研究预测推演工具**。它不是点预测模型，而是一个**态度动力学 / 意见扩散 / 采纳扩散的沙盒**——让你在投入真实预算前，先在"用户研究基座"上反复推演：新品会不会卖、社媒活动会不会爆、品类机会成不成立。

> 本页面为**研究 / 内部工具原型**，仿真数据由前端确定性算法生成，仅供理解推演逻辑与机制之用，不构成任何销售、财务或投资建议。MiroFish 原始项目采用 AGPL-3.0 许可证，由盛大集团战略孵化。

## 在线演示

部署后访问 GitHub Pages 地址（仓库 `mirofish-console` 的衍生版本）。

## 本地运行

单文件 `index.html`，ECharts 走 CDN，无构建步骤。两种方式：

- 直接双击 `index.html` 在浏览器打开；
- 或本地起服务（推荐，避免个别浏览器对 `file://` 的限制）：

```bash
cd user-research-console
python3 -m http.server 8080
# 浏览器打开 http://localhost:8080/index.html
```

## 三个预测场景

| 场景 | 回答的核心问题 | 干预变量（上帝视角） |
|---|---|---|
| 新品上市预测 `product` | 首发转化、口碑与用户反馈如何演化？哪些分群会差评反弹？ | 卖点透明度 / 痛点解决力度 / 达人·口碑投放 |
| 社媒活动预测 `social` | 内容会不会爆 / 会不会翻车？哪个分群放大传播？KOL 投在哪类？ | 内容契合度 / 互动·激励力度 / KOL 投放强度 |
| 品类机会预测 `category` | 机会点是否成立？哪类需求最值得切入？品类结构怎么优化？ | 机会清晰度 / 需求覆盖度 / 生态·供给准备 |

## 用户基座（核心）

仿真的人格来源不是虚构角色，而是从你的用户研究体系中聚类出来的**真实用户分群**。当前内置 8 个示例分群（尝鲜型 / 价格敏感 / 品质党 / 观望 / 痛点击中 / KOL·达人 / 差评传播 / 竞品用户），每个分群带：

- **态度 x**（-1 抵触·不买 ↔ +1 接受·种草·购买意向）
- **价格—品质取向 y**（-1 价格敏感 ↔ +1 品质/体验导向）
- **活跃度**、**是否倡导者/KOL**、**基座占比**

### 如何接入你真实的用户数据

把 VOC 聚类结果 / 类目洞察分群，按下方 schema 替换即可——**引擎逻辑不变，换的是"人格来源"**：

```json
{
  "segments": [
    {
      "name": "价格敏感者",
      "weight": 0.18,
      "x": -0.15,
      "y": -0.70,
      "act": 0.55,
      "advocate": false,
      "tags": ["用户画像:价格敏感", "动机:性价比", "阻碍:预算有限"]
    }
  ]
}
```

接入映射直接对齐你已经建好的两套工具：

- **voc-report**（VOC 九大标签模块）：用户画像分群 / 购买动机 / 使用场景 / 使用阻碍 / 忠诚度 / 改进建议 / 十三种需求 → 决定每个分群的 x、y 与痛点。
- **category-insight-hub**（类目洞察中心）：市场细分、JTBD、需求四象限、价格带 → 决定分群构成、需求轴与干预杠杆。

## 数值引擎逻辑（什么真 / 什么近似）

引擎是一个**确定性 Agent-Based Model（ABM）**，用种子化伪随机数（mulberry32）保证同参数可复算。忠实实现了 OASIS 的真实机制：

- 激活（事件初期更亢奋）→ 推荐（Reddit hot-score + 二维嵌入相似度，制造信息茧房）→ 动作空间决策（like/repost/comment/create_post/follow…）→ 有界信心 + DeGroot 社会影响的观点更新 → 记忆统计 + 帖子老化。
- 三种网络拓扑可选（无标度 / 小世界 / 随机）。
- **蒙特卡洛模式**：同网络、不同随机流复算 N 次，输出均值 ± 95% 置信区间与终态分布，让结论带统计显著性。

**被替代的部分**：真实 MiroFish 里动作决策与发帖文本由 LLM（Qwen-plus / GPT-4o）生成，此处用基于立场差 + 人格特质的**规则化策略**近似——保留了决策的逻辑结构，去掉了需要大模型的语言生成与后端。这也是它能纯前端、零成本运行的原因。

## 技术栈

- 前端：原生 HTML / CSS / JavaScript（Radium 暗色玻璃拟态风格）
- 可视化：ECharts（CDN）
- 仿真：浏览器内确定性 ABM，无后端、无外部 API
